Chapter 1

What is Hier?

Hier is a systems language — it compiles to native machine code, like C or Rust. But its syntax looks like Python:

hier
fn greet(name: string) -> string:
    return "hello " + name + "\n"

fn main():
    print("what is your name: ")
    name := input()
    print(greet(name))

Three things make Hier different from the languages you know:

ConceptJS / PythonRustHier
MemoryGarbage collectorBorrow checker + lifetimesImplicit arenas (automatic)
TypesDynamicStatic, inferredStatic, inferred
Nullnull / NoneOption<T>Option(T)
ErrorsExceptionsResult<T,E>Result(T, E)
ConcurrencyShared mutable stateSend/Sync + lifetimesCopy-in/copy-out (race-free)
💡
The Key Insight: Value Semantics
Every value is independently owned. When you assign b = a, b gets its own deep copy. Two variables never share storage. This lets the compiler manage memory automatically — each scope gets an arena (a memory pool) that frees everything at once when the scope ends. No free(), no GC, no lifetimes.
❓ Check Your Understanding

1. How does Hier manage memory?

Garbage collection, like JavaScript
Manual malloc/free, like C
Implicit arenas — the compiler inserts all allocation and reclamation
Borrow checker with lifetimes, like Rust
Correct! Hier uses lexical arenas — one per scope — that free everything at once when the scope exits. The programmer never sees an arena API.
Not quite. Hier uses implicit hierarchical arenas — one per scope, freed automatically. No GC, no lifetimes, no manual free.

hier

Chapter 4

Functions

hier
# returns an int
fn add(a: int, b: int) -> int:
    return a + b

# returns nothing (no -> type)
fn say(s: string):
    print(s + "\n")

# inout: mutate the caller's variable
fn incr(n: inout int):
    n = n + 1

fn main():
    print(str(add(3, 4)) + "\n")   # 7
    x := 10
    incr(&x)                  # x is now 11
    print(str(x) + "\n")    # 11

Web Developer Mental Model

  • fn foo(x: int) — like passing a primitive in JS. The function gets its own copy.
  • fn foo(xs: [int]) — like a read-only view. You can read but not mutate.
  • fn foo(xs: inout [int]) — like passing by reference. The function can mutate the caller's array. Mark the call with &.

String Interpolation

Like template literals in JS:

hier

age := 36
print(f"{name} is {age}\n")           # Ada is 36
print(f"2 + 3 = {2 + 3}\n")          # 2 + 3 = 5

Use f"..." to interpolate. {expr} inserts the value. Plain "..." strings are never interpolated.

Chapter 6

Strings

Strings are immutable byte buffers. You build new ones with +.

hier
s := "hello"
print(str(len(s)) + "\n")         # 5
print(str(s[0]) + "\n")           # 104 (byte value of 'h')
s2 := s + " world"               # "hello world"
print(substr("hello world", 0, 5)) # "hello"
print(str(find(s2, "world")))   # 6
parts := split("a,b,c", ",")      # ["a", "b", "c"]

The Secret Optimization: In-Place Append

⚡
O(n) string building, automatically
When you write total = total + x in a loop, the compiler grows total's buffer in place — O(n) total, not O(n²). This is automatic: the source just looks like normal string concatenation.
hier
fn main():
    total := ""
    for i in range(1, 6):
        total = total + str(i)        # O(n), not O(n^2)!
        if i < 5:
            total = total + ", "
    print(total + "\n")           # "1, 2, 3, 4, 5"

The char Type

char is a single byte. Use 'x' syntax. Appending a char to a string is a zero-alloc byte push:

hier
s := "hello"
s = s + '\n'             # zero-alloc byte append
s = s + ('0' + 5)          # appends '5' (char arithmetic)

Chapter 7

Arrays

Arrays are dynamic, heap-allocated, value-semantic lists. Assigning one makes a deep copy.

hier
xs := [10, 20, 30]           # [int] literal
empty := []int               # empty (type required)
push(xs, 40)                 # [10, 20, 30, 40]
last := pop(xs)              # 40
print(str(len(xs)))           # 3
xs[0] = 99                   # write (bounds-checked)

# VALUE SEMANTICS: assignment is a deep copy
ys := xs                      # independent copy
ys[0] = 0                    # never touches xs
⚠️
Array parameters are read-only borrows
A [int] parameter is a borrow — no copy, but you can't mutate it. Copy first (b := a) to get a mutable local, or take it inout.

Slices

xs[a:b] is a sub-range. Passed to a read-only parameter: zero-copy view. Stored in a variable: deep copy.

hier
xs := [10, 20, 30, 40, 50]
print(str(sum(xs[1:4])))     # 90 — zero-copy view
mid := xs[1:4]                # deep copy: [20, 30, 40]

Returning Arrays

hier
fn make_squares(n: int) -> [int]:
    r := []int
    for i in range(n):
        push(r, i * i)
    return r              # promoted into caller's arena

Chapter 8

Structs

Structs are named product types with value semantics. Think plain objects that are always deep-copied.

hier
struct Point:
    x: int
    y: int

struct Rect:
    lo: Point
    hi: Point

fn area(r: Rect) -> int:
    return (r.hi.x - r.lo.x) * (r.hi.y - r.lo.y)

fn main():
    a := Point(1, 2)              # positional construction
    r := Rect(a, Point(4, 6))
    print(str(area(r)) + "\n")   # 9
    b := a                         # deep copy
    b.x = 99                      # never touches a
💡
Structs with heap fields
A struct can own strings, arrays, and other structs. On assignment, everything is deep-copied — the copy is independent at every nesting level.

Chapter 10

Option and Result

Option(T) — the no-null story

Instead of null, Hier uses Option(T): either Some(value) or None.

hier
fn index_of(xs: [int], target: int) -> Option(int):
    for i in range(len(xs)):
        if xs[i] == target:
            return Some(i)
    return None

fn main():
    match index_of([10, 20, 30], 20):
        Some(i):
            print("found at " + str(i) + "\n")
        None:
            print("not found\n")
⚠️
The match is exhaustive — you must handle both arms. The compiler won't let you forget the null check.

Result(T, E) — the no-exceptions story

hier
fn checked_div(a: int, b: int) -> Result(int, string):
    if b == 0:
        return Err("divide by zero")
    return Ok(a / b)

# or_return: unwrap or propagate the error
fn add_two(a: string, b: string) -> Result(int, string):
    x := parse_digit(a) or_return  # unwrap or return Err
    y := parse_digit(b) or_return
    return Ok(x + y)

Chapter 11

Enums and Pattern Matching

An enum is a tagged union — a value that is exactly one of several variants, each carrying data. Think of it as a type-safe switch on steroids.

hier
enum Shape:
    Circle(float)
    Rect(float, float)

fn area(s: Shape) -> float:
    match s:
        Circle(r):
            return 3.14159 * r * r
        Rect(w, h):
            return w * h

Recursive Enums (Trees / ASTs)

hier
enum Expr:
    Num(int)
    Add(Expr, Expr)
    Mul(Expr, Expr)

fn eval(e: Expr) -> int:
    match e:
        Num(v):
            return v
        Add(l, r):
            return eval(l) + eval(r)
        Mul(l, r):
            return eval(l) * eval(r)

fn main():
    # (2 + 3) * 4 = 20
    expr := Mul(Add(Num(2), Num(3)), Num(4))
    print(str(eval(expr)) + "\n")  # 20
💡
Every match must be exhaustive — every variant needs an arm. The compiler won't let you forget a case. Enums are value-semantic too: copying one deep-copies the whole tree.

Chapter 14

Value Semantics: The Mental Model

🎯
This is the most important concept in Hier. Internalize these three rules and everything else follows.

Rule 1: Assignment copies

hier
a := [1, 2, 3]
b := a              # independent copy
push(b, 4)
print(str(len(a)))  # 3 — a is unchanged

Rule 2: Parameters are copies (unless inout)

A [int] parameter is a read-only borrow. To mutate, copy first or use inout.

Rule 3: No two variables ever share storage

hier
a := Person("Ada", ["x"])
b := a               # deep copy
b.name = "Alan"
b.tags[0] = "y"
# a.name is still "Ada", a.tags[0] is still "x"

Why This Matters

Because every value has exactly one owner, the compiler can attach each value to a scope. When the scope ends, all its values are freed at once — no GC, no refcounts, no free(). A loop running a million iterations does so in constant memory because each iteration's scratch arena resets.

❓ Check Your Understanding

1. Can two variables share storage in Hier?

No — every assignment makes an independent deep copy
Yes — arrays and strings are shared by default
Only through explicit reference types
Correct! This is value semantics: two variables never share storage. Every assignment deep-copies. This is what makes implicit arenas sound.
No two variables ever share storage. Every assignment deep-copies. This is the core of value semantics and what makes implicit arenas possible.

Chapter 16

Methods (UFCS)

x.foo(a, b) is sugar for foo(x, a, b) — Uniform Function Call Syntax. No classes, no self.

hier
struct Vec3:
    x: float
    y: float
    z: float

fn vadd(a: Vec3, b: Vec3) -> Vec3:
    return Vec3(a.x + b.x, a.y + b.y, a.z + b.z)

fn vlen(v: Vec3) -> float:
    return sqrt(v.x * v.x + v.y * v.y + v.z * v.z)

fn main():
    a := Vec3(1.0, 2.0, 3.0)
    b := Vec3(4.0, 5.0, 6.0)
    c := a.vadd(b)        # sugar for vadd(a, b)
    print(str(c.vlen()))  # sugar for vlen(c)

Calls chain naturally: a.vadd(b).vlen() = vlen(vadd(a, b)).

Chapter 17

Concurrency

Hier's concurrency is race-free by construction. Every value crossing a thread boundary is deep-copied. No locks, no lifetimes.

spawn / wait

hier
fn fib(n: int) -> int:
    if n < 2: return n
    return fib(n-1) + fib(n-2)

fn main():
    t1 := spawn fib(30)       # runs on another thread
    t2 := spawn fib(28)       # in parallel
    a := wait(t1)              # blocks, result deep-copied back
    b := t2.wait()             # alternative syntax

parallel for

hier
total := 0
parallel for i in range(1000000):
    total += score(i)          # reduction: safe

Channels

hier
ch := channel(string, 16)     # bounded lock-free queue
ch.send("hello")            # deep-copies in
match recv(ch):                # deep-copies out
    Some(s): ...
    None: ...                  # closed and drained
close(ch)                       # signal no more sends
❓ Check Your Understanding

1. How does Hier prevent data races?

A borrow checker rejects unsafe code at compile time
Every value crossing a thread boundary is deep-copied
Runtime locks are automatically inserted
Correct! The copy-in/copy-out convention means a task shares zero bytes with its spawner. Race freedom falls out of value semantics.
Hier uses copy-in/copy-out: arguments are deep-copied into the task, results are deep-copied back. No shared state = no races. No borrow checker needed.

Memory Model Rules

  1. Every scope gets its own arena — functions, blocks, loop iterations.
  2. Assignment copies — b = a gives b an independent deep copy.
  3. Parameters are borrows (read-only) — copy first or use inout to mutate.
  4. Return values are promoted into the caller's arena — no dangling pointers.
  5. Loop scratch arenas reset each iteration — bounded memory in loops.
  6. The programmer never sees any of this — no malloc, free, or arena API.
