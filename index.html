Implicit hierarchical arenas,
under value semantics.

Tycho is a research project built to test one idea. Each scope owns a memory arena, freed when the scope exits; with no reference type in the language, the compiler sees every value's lifetime from the syntax and inserts every allocation and free itself. Automatic memory management from lexical scope, not a runtime — and no manual `free`. 

fn greet(name: string) -> string:
    return "hello " + name

fn main():
    name := input()
    println(greet(name))

the idea

Lifetimes you can see in the source.

The arena is an old, fast idea — bump-allocate, free everything at once when the scope ends. The catch was always knowing when a value may outlive its arena; with pointers, that needs whole-program alias analysis.

Value semantics removes the question. There is no reference type: b := a copies, and you cannot store or return a pointer into another value's memory. So a value can leave a scope in exactly two ways, both visible in the source:

↓ downpassed as an argument to a callee.
↑ upreturned to the caller.

That's the whole thesis — the compiler places every allocation from the syntax, with no whole-program analysis and no annotations to write. Two optimizations keep it fast, both sound because the value is provably un-aliased: a returned value is built in the caller's arena (so return is a move, not a copy), and acc = acc + x in a loop grows one buffer in place — the textbook O(n²) string build becomes O(n). Neither is visible in your code.

the evidence

Experimental in scope, not in rigor.

It's a research project — an experiment testing one idea — but a heavily-checked one, which is the part most experiments this young skip.

self-hostTwo compilers: a reference in C, and a second one written in Tycho that compiles itself on the same implicit arenas it emits. make fixpoint holds it to reproducing its own emitted C byte-for-byte and matching the reference on every test.
fuzzA differential fuzzer runs both compilers on random programs under ASan/UBSan; every example is built twice — native and sanitized — and checked against a committed golden.
localNo cloud CI to take on faith. make ci runs the whole gate on your machine.

Where the model helps and where it costs are both measured — as machine-specific ratios, with the workloads that produce them — in the performance notes. Pointer-shaped data (graphs, trees) is where value semantics is least free; the docs are honest about that, not just about the wins.

the language

Small, Go/Odin-shaped, honest about what it is.

Python/Nim-flavored syntax, Go/Odin-like semantics; the value-semantics core comes from Hylo.

  • Value semantics — no aliasing, no GC
  • Generics — $T, monomorphized
  • Sum types — Option/Result, match
  • Methods (UFCS) — no classes
  • Closures — value-capture
  • Concurrency — spawn, channels, parallel-for
  • FFI to C — extern fn, handles, bytes
  • Packages + corelib — import, 36 modules

try it

A C compiler and make — that's the whole toolchain.

# a `cc` and `make`, nothing else
$ git clone https://github.com/StefanVonRanda/tycho
$ cd tycho
$ make                                  # builds ./tychoc
$ ./tychoc examples/hello.ty && ./examples/hello
