◇ an experimental systems language · transpiles to C

Automatic memory management —
no garbage collector, no manual free.

Tycho is a research project testing one idea: implicit hierarchical arenas under value semantics. Each scope owns a memory arena, freed when the scope exits. Because the language has no reference type, the compiler can see every value's lifetime from the syntax alone, and inserts every allocation and free for you — the memory management comes from lexical scope, not from a runtime. It transpiles to C, and builds with a plain cc and make. 

fn greet(name: string) -> string:
    return "hello " + name

fn main():
    name := input()
    println(greet(name))

how it works

Every scope owns a memory arena.

An arena is an old, fast idea: hand out memory by bumping a pointer, then free it all at once when the scope ends. The catch has always been knowing when a value is allowed to outlive its arena — with pointers, answering that needs whole-program alias analysis, and that's the hard part.

Value semantics removes the question. Tycho has no reference type: b := a copies, and you can't store or return a pointer into another value's memory. So a value can leave a scope in only two ways, and both are right there in the source:

↓ downpassed as an argument into a function you call.
↑ upreturned back out to your caller.

That's the whole thesis. With only those two exits, the compiler places every allocation straight from the syntax — no whole-program analysis, and no lifetime annotations for you to write.

Two optimizations keep value semantics from being slow, both sound because a value is never aliased. A value you build and return is allocated straight into the caller's arena, so return moves it instead of copying. And acc = acc + x in a loop grows one buffer in place — the textbook O(n²) string build becomes O(n). Neither shows up in your code.

the language

What the language looks like.

The syntax is Python/Nim-flavored and the semantics are Go/Odin-like; the value-semantics core comes from Hylo. What's in the box:

  • Value semantics — no aliasing, no GC
  • Generics — $T, monomorphized
  • Sum types — Option/Result, match
  • Methods (UFCS) — no classes
  • Closures — value-capture
  • Concurrency — spawn, channels, parallel-for
  • FFI to C — extern fn, handles, bytes
  • Packages + corelib — import, 36 modules

how it's checked

An experiment, but a heavily-checked one.

Tycho is a research project — an experiment testing one idea — but heavily checked, which is the part most experiments this young skip. It's experimental in scope, not in rigor.

self-hostThere are two compilers: a reference written in C, and a second one written in Tycho that compiles itself, running on the same implicit arenas it emits. make fixpoint holds it to reproducing its own emitted C byte-for-byte, and to matching the C reference on every test.
fuzzA differential fuzzer runs both compilers on random programs under ASan/UBSan; every example is built twice — native and sanitized — and checked against a committed golden output.
localThere's no cloud CI to take on faith. make ci runs the whole gate on your own machine.

Where the model helps and where it costs are both measured — as machine-specific ratios, with the workloads that produce them — in the performance notes. Pointer-shaped data like graphs and trees is where value semantics is least free, and the docs are honest about that, not just about the wins.

try it

A C compiler and make — that's the whole toolchain.

# a `cc` and `make`, nothing else
$ git clone https://github.com/StefanVonRanda/tycho
$ cd tycho
$ make                                  # builds ./tychoc
$ ./tychoc examples/hello.ty && ./examples/hello
